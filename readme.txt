# 프로젝트 요약 (공유용)

## 1) 프로젝트 한 줄 정의

**필라테스 강사용 수업 기록 & 회원 피드백 SaaS**
(웹 기반 서비스가 본체, 모바일 앱은 추후 옵션)

## 2) 문제 정의

현재 강사들의 수업 후 커뮤니케이션/기록은 보통 아래로 분산되어 있음:

* 카카오톡/DM(흩어짐, 검색 어려움, 체계 없음)
* 노트/메모앱(회원에게 전달 불편)
* 머릿속 기억(누적 불가, 강사 바뀌면 단절)

결과:

* 회원은 “오늘 뭐 했지?”가 남고, 관리받는 느낌이 약함
* 강사는 수업 퀄리티를 기록/축적하지 못해 재방문·유지에 불리
* 센터/강사는 체계적 리포트가 없어서 차별화가 어려움

## 3) 해결책(제안)

**수업 종료 후 1~2분 내 작성 가능한 “레슨 리포트”를 생성하고**
회원에게 **정리된 형태로 전달**하며
해당 기록이 회원별 히스토리로 **자동 누적**되도록 한다.

핵심 가치:

* 강사: 기록 자동화 + 회원관리(관계/유지) + 전문성 증명
* 회원: 내가 어떤 상태인지, 무엇이 부족한지, 다음 목표가 뭔지 “가시화”
* 센터(확장 시): 강사 품질/회원 만족 관리 가능

---

# 포지셔닝(중요)

## “회원관리 앱”이 아니라 무엇인가?

* ❌ 센터 운영자용 예약/결제/출석 중심 “관리 프로그램”과 정면 경쟁하지 않는다.
* ✅ 강사가 직접 쓰는 **수업 품질 기록 + 회원 피드백 전달 툴**이다.

권장 명칭(외부 커뮤니케이션):

* “강사용 수업 리포트/피드백 서비스”
* “강사 전용 레슨 기록 SaaS”
* “Instructor Console / Coach Portal”

# 대상 고객 & 수익화

## 1) 1차 타겟

* 프리랜서 강사
* 1인 스튜디오 운영 강사
* 소규모 센터의 대표 강사 (센터 시스템과 무관하게 개인이 사용 가능해야 함)

## 2) 과금 모델(초안)

* 구독형(SaaS): 강사 1인 기준 월 결제
* 회원 수 기준 티어(예: 20명/50명/100명)
* 고급 기능(추후): 템플릿/자동요약/리포트 디자인/브랜딩/센터 계정

# MVP 범위(“지금 당장 팔 수 있는 최소 기능”)

## MVP 목표

**“수업 끝나고 리포트 작성 → 회원에게 링크/메시지로 전달 → 기록 누적”**
이 흐름이 매끄럽게 돌아가면 MVP 성립.

## MVP 기능 목록 (강사용)

1. 로그인/회원 목록
2. 회원 상세(통증/주의사항/최근 리포트)
3. 수업 세션 생성(간단) 또는 캘린더형 세션 선택(추후)
4. **수업 리포트 작성(템플릿 기반, 1~2분 완성)**
5. **회원에게 리포트 공유(링크 + 메시지 템플릿)**
6. 리포트 히스토리 누적/검색(회원별)

## MVP에서 “안 하는 것”(중요)

* 결제/예약/환불/스케줄링 풀세트 (시장에 이미 많음)
* 복잡한 센터 권한/정산/CRM
* 앱스토어 배포 (웹 SaaS로 충분)

# 사용자 흐름(시퀀스 중심)

## 시퀀스 1: 수업 후 리포트 생성 & 공유 (핵심)

1. 강사 로그인
2. 회원 선택(또는 오늘 세션 선택)
3. 리포트 템플릿 열기
4. 체크/짧은 메모로 작성
5. 저장
6. “공유” 클릭 → 회원에게 전달할 링크/메시지 생성
7. 회원은 링크로 리포트 열람 (로그인 없이 or 간편 인증)

### 결과

* 강사는 “기록”을 남기고
* 회원은 “관리받는 경험”을 받음
* 데이터는 누적되어 다음 수업 품질이 올라감

## 시퀀스 2: 회원 히스토리 기반 수업 준비(강사 락인)

1. 강사 → 회원 상세 진입
2. 최근 3회 리포트 요약/통증 플래그 확인
3. 오늘 목표/주의사항 확인
4. 수업 진행

# 데이터 모델(초안)

## 핵심 엔티티

* Instructor(강사)
* Client(회원)
* Session(수업 세션: 날짜/강사/회원/형태[개인/그룹])
* Report(리포트: 세션 기반, 텍스트/체크리스트/추천과제)
* ReportShare(공유: 링크 토큰, 만료, 조회 로그)
* Flags(통증/주의사항: 어깨/무릎/허리/측만 등)
* Template(리포트 템플릿: 섹션/문항)

## 리포트 기본 구성(템플릿 예시)

* 오늘 진행한 내용(체크리스트 + 짧은 메모)
* 잘 된 점(Strength)
* 부족한 점(Need to improve)
* 다음 목표(Next goal)
* 홈워크/주의사항(Optional)
* 통증/불편감 변화(Optional)

# API 설계(초안)

## 인증

* POST /auth/login
* POST /auth/refresh
* GET /me

## 회원

* GET /clients
* POST /clients
* GET /clients/{clientId}
* PATCH /clients/{clientId}

## 세션(간단 버전)

* POST /sessions (clientId, date, type)
* GET /sessions?date=YYYY-MM-DD
* GET /sessions/{sessionId}

## 리포트

* POST /reports (sessionId, templateId, payload)
* GET /reports/{reportId}
* GET /clients/{clientId}/reports?limit=20
* PATCH /reports/{reportId}

## 공유

* POST /reports/{reportId}/share  -> shareUrl/token 생성
* GET /share/{token} -> 회원 열람 페이지 (권한/만료 검사)
* POST /share/{token}/feedback (추후: 회원 피드백)

# 기술 구조(권장)

## 형태

* **웹 기반 SaaS**
* Frontend(Web) + Backend API + DB

## 핵심 요구

* 모바일에서도 잘 되는 반응형 웹(강사가 수업 후 폰으로 쓰기 때문)
* 공유 링크는 외부에서 열려야 함(회원 편의)
* 데이터 보호(링크 토큰, 만료, 접근 로그)

## 구현 후보(예시)

* Front: Next.js/React
* Back: Node/NestJS or Spring Boot (개발 선호에 맞춤)
* DB: PostgreSQL
* Auth: JWT + Refresh
* Share: Signed token + 만료 + 조회 기록

---

# 경쟁/차별화 관점(요약)

* “예약/결제/출석”이 아니라 **수업 가치 전달(리포트)과 기록 누적**이 코어
* 강사가 개인적으로 구독 가능해야 함(센터 도입 장벽 제거)
* 템플릿과 UX가 승부처: “1~2분 작성”이 핵심 KPI

---

# MVP 성공 지표(KPI) 초안

* 리포트 작성 소요 시간: 평균 2분 이내
* 주간 활성 강사 비율(WAU): 작성/공유가 반복되는지
* 회원 리포트 열람률: 공유 후 열람 퍼센트
* 리포트 누적률: 회원당 최근 4주 리포트 개수
* 재구독/이탈률
